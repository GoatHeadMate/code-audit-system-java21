# 代码审计系统 —— 全流程与技术说明

> 适用分支:`master`(JavaParser 白盒 + Claude Agent SDK 监督者/子代理)
> 一句话定位:**用静态分析(JavaParser)把"入口→危险 sink"的候选攻击路径挖出来当线索,再交给 Claude 监督者带 8 类子代理逐条复核确认,产出带数据流路径的漏洞报告。**

---

## 1. 端到端流程(分阶段)

### 阶段 A:提交与源码准备

1. **前端提交**([index.html](src/main/resources/static/index.html)):用户上传 ZIP 或填 Git URL,可选附带"外部接口清单";可先调 `/audit/interfaces` 扫描接口、勾选后再 `/audit/{jobId}/start` 启动。
2. **接收建任务**([AuditController](src/main/java/com/huawei/audit/api/AuditController.java)):落盘上传文件,`AuditJobStore` 建 `AuditJob`,返回 jobId;前端用 `/audit/{jobId}/logs`(SSE)订阅实时日志。
3. **源码准备**([AuditOrchestratorImpl.run](src/main/java/com/huawei/audit/orchestrator/impl/AuditOrchestratorImpl.java)):`SourceWorkspaceService` 解压 ZIP / 克隆 Git 到 `workspace/audit_<id>/project`。

### 阶段 B:技术画像与 Hunter 选型

4. **技术画像**(`TechProfileScanner`):解析得到 `primary_language / build_tool / web_framework / orm / dependencies / app_package`。
5. **Hunter 调度**([HunterScheduler](src/main/java/com/huawei/audit/hunter/HunterScheduler.java)):从 8 类 hunter 中选候选,并按技术画像**加权前置**:
   - 默认必上:`code_execution`、`authorization`
   - 命中 MyBatis/JPA/Hibernate → 加 `sql_injection`
   - 命中 fastjson/xstream/snakeyaml → 加 `unsafe_parsing`
   - 命中 h2/log4j/actuator → 加 `component_vulns`
   - 完整 8 类:`code_execution / sql_injection / unsafe_parsing / ssrf / file_operations / http_output / authorization / component_vulns`

### 阶段 C:langgraph4j 状态图编排

6. **编排图**([IntelligentAuditGraph](src/main/java/com/huawei/audit/orchestrator/IntelligentAuditGraph.java)):一张三节点状态图 `START → prepare_evidence → run_supervisor → finalize → END`。

### 阶段 D:白盒静态分析(prepare_evidence 节点,核心)

7. **静态分析管线**([WhiteBoxAnalysisServiceImpl.analyze](src/main/java/com/huawei/audit/analysis/impl/WhiteBoxAnalysisServiceImpl.java)),顺序为:
   1. **入口发现**(`EntryPointDiscoverer` 多实现:Servlet / Async / WebFlux,均基于 JavaParser AST)→ 发现 HTTP/WS 入口。
   2. **源码索引**([JavaSourceIndexer](src/main/java/com/huawei/audit/analysis/impl/JavaSourceIndexer.java)):JavaParser 解析全部 `.java`,抽出方法节点、调用点、危险 sink(`DangerousSinkClassifier`)。**可选挂载 SymbolSolver** 精确解析调用类型(详见 [SYMBOL_SOLVER_说明.md](SYMBOL_SOLVER_说明.md))。
   3. **补充 sink 源**:`MyBatisXmlScanner`(Mapper XML 拼接 SQL)、`ConfigTemplateScanner`(配置/模板)。
   4. **LLM 辅助 sink**:`LlmSinkExpander`(按依赖让 Claude 补第三方危险 API 规则)、`SuspiciousCallReviewer`(让 Claude 复核可疑调用是否为 sink)。
   5. **调用图**([CallGraphBuilder](src/main/java/com/huawei/audit/analysis/impl/CallGraphBuilder.java)):用调用点的 `receiverType` 解析每条调用边的目标方法(含接口→实现)。
   6. **传递性 sink**([TransitiveSinkResolver](src/main/java/com/huawei/audit/analysis/impl/TransitiveSinkResolver.java)):沿调用图把 sink 分类向上游传播。
   7. **入口绑定**(`EntryPointBinder`):把发现的入口绑定到已索引的方法节点。
   8. **候选路径**([CandidatePathFinder](src/main/java/com/huawei/audit/analysis/impl/CandidatePathFinder.java)):在调用图上做"入口→sink"有界 BFS(先用 `ReverseReachability` 算距离剪枝),产出 `candidate_paths`。
   9. **污点校验**(`MethodTaintSummarizer` + `TaintFlowVerifier`):对候选路径做方法级污点摘要与校验。
   10. **存储型/异步链**(`StorageWritePathFinder` + `StoredCandidateCorrelator`):识别"先写存储、后异步执行"的两段式攻击链。
   11. **覆盖度**(`CoverageCalculator`):产出 `entrypoints/methods/sinks/candidate_paths/unresolved_calls` 等指标(就是日志里那行 `[whitebox] ...`)。
8. **证据打包**([EvidencePreparationServiceImpl](src/main/java/com/huawei/audit/agent/impl/EvidencePreparationServiceImpl.java)):把候选路径/存储候选**分块**写成 `evidence/whitebox/*.json`,按 hunter 相关性拆成**每个 hunter 的证据包**,并由 `SubagentDefinitionService.materialize` 生成每个子代理的 **Claude Skill**(判定知识 + 复核契约 `REVIEW_CONTRACT`)。

### 阶段 E:监督者 + 子代理复核(run_supervisor 节点)

9. **Java 侧调用**:`SupervisorAgent` → [ClaudeSidecarClient.supervise](src/main/java/com/huawei/audit/agent/ClaudeSidecarClient.java),POST `/v1/supervise`,用 `HttpResponse.BodyHandlers.ofLines()` 接收 **NDJSON 流**(逐行事件:子代理启动/完成/助手文本/最终结果)。
10. **Python sidecar 侧**([sdk_runner.py](sidecar/src/claude_audit_sidecar/sdk_runner.py)):
    - 用 **Claude Agent SDK** 起 **1 个监督者 session**,预置 8 个**原生子代理**;
    - 以原生 **`/goal` 命令**驱动监督者跑到"完成条件满足"才结束(完成条件放 user prompt、≤4000 字符;完整审计任务放 system prompt,见下文修复记录);若环境不暴露 `/goal` 则**自动回退**普通 prompt;
    - 子代理只给只读工具(`Read/Glob/Grep/Agent/SendMessage`),禁用 `Bash/Write/Edit/...`,读证据包 + 回源码确认;
    - 加载 **Claude Skills** 作为子代理领域判定知识;
    - 返回唯一 JSON 信封:`{ selected_hunters, rationale, findings }`。

### 阶段 F:汇总产出(finalize 节点)

11. **去重 + 攻击链关联**:`FindingDeduplicator` 去重,`AttackChainCorrelator` 跨发现合成攻击链(如"越权 + 命令注入 → 未认证 RCE")。
12. **回写**([AuditOrchestratorImpl](src/main/java/com/huawei/audit/orchestrator/impl/AuditOrchestratorImpl.java)):`findings.json` + `stats` + `task_summary` 落盘并写回任务,前端三个标签页(实时日志 / 漏洞发现 / 摘要)展示。

## 2. 技术栈清单(每项技术 + 它在系统里的作用)

### 后端 / 编排

| 技术 | 作用 |
|---|---|
| **Java 21** | 后端语言;用 **虚拟线程**(`spring.threads.virtual.enabled=true`)承载并发任务与子代理,降低线程成本 |
| **Spring Boot** | Web 容器、依赖注入、配置(`application.yml`)、REST 控制器、`ExecutorService` 任务执行 |
| **SSE (Server-Sent Events)** | `/audit/{id}/logs` 单向推送实时审计日志到前端(`JobLogBroker` + `SseEmitter`) |
| **langgraph4j** | **状态图编排框架**:把"准备证据→跑监督者→汇总"建成可组合的有向图节点(`StateGraph`/`CompiledGraph`),管理节点间状态传递 |
| **langchain4j** | LLM 集成框架基座(智能编排开关 `intelligent_orchestrator`),与 langgraph4j 配合驱动审计流程 |
| **Jackson** | 全链路 JSON 序列化(证据包、findings、与 sidecar 的请求/响应) |
| **Maven** | 构建与依赖管理;`spring-boot-maven-plugin` 启动应用(实验性 JVM 参数也挂在这里) |

### 静态分析(白盒)

| 技术 | 作用 |
|---|---|
| **JavaParser** | **纯 AST 解析器**:解析全部 `.java`,抽取方法、调用点、注解、入口;入口发现(Servlet/Async/WebFlux)与索引层全部基于它 |
| **JavaSymbolSolver**(可选) | **类型/符号解析**:精确算出调用接收者与参数的真实类型(含接口→实现),把调用图从"几乎连不通"打通成丰富的候选路径;默认关、`-Daudit.symbol-solver=true` 开。详见 [SYMBOL_SOLVER_说明.md](SYMBOL_SOLVER_说明.md) |
| **CallGraphBuilder / CandidatePathFinder / ReverseReachability** | 自研调用图构建 + 入口→sink 有界 BFS + 距离剪枝,产出候选攻击路径 |
| **DangerousSinkClassifier / TransitiveSinkResolver** | 16+ 类危险 sink 模式识别 + 沿调用图传播 sink 分类 |
| **MethodTaintSummarizer / TaintFlowVerifier** | 方法级污点摘要与候选路径污点校验(请求值是否可控、是否经净化) |
| **MyBatisXmlScanner / ConfigTemplateScanner** | 补充非 Java 源的 sink(Mapper XML 拼接 SQL、配置/模板注入) |

### LLM / 智能体

| 技术 | 作用 |
|---|---|
| **Claude Agent SDK**(Python) | 跑 **1 个监督者 + 8 个原生子代理** 的多智能体会话;子代理只读复核静态分析给出的候选路径 |
| **`/goal` 原生命令** | 让监督者**目标驱动**地跑到"完成条件满足"才结束(委派子代理→复核→产出唯一 JSON);不可用时自动回退普通 prompt |
| **Claude Skills** | 每个子代理的**领域判定知识**(如何判命令注入/越权等),由 `SubagentDefinitionService` 按 job 物化到 `.claude/skills/` 供 CLI 发现 |
| **FastAPI + uvicorn** | Python sidecar 的 HTTP 服务端,暴露 `/v1/query`、`/v1/supervise` |
| **HTTP NDJSON 流** | Java↔Python 传输:监督者过程以**逐行 JSON 事件**流式回传(子代理启动/完成/文本/结果),前端实时可见 |
| **Pydantic** | sidecar 侧请求/事件的类型化模型与校验 |

### 前端

| 技术 | 作用 |
|---|---|
| **原生 HTML + JavaScript** | 单页应用,无框架;管理上传、接口选择弹窗、任务列表、日志/漏洞/摘要渲染 |
| **TailwindCSS(CDN)** | 全部样式(实用类),含严重度配色、攻击路径时间线等 |
| **EventSource(SSE 客户端)** | 订阅 `/audit/{id}/logs`,实时渲染监督者/子代理事件 |

## 3. 关键设计点

1. **静态分析当"线索引擎",LLM 当"判定引擎"**:JavaParser 负责把入口→sink 的候选路径挖出来(高召回、可能有误报),Claude 子代理负责回源码确认/剔除误报(高精度)。两者职责分离。
2. **SymbolSolver 是召回放大器**:关闭时候选路径极稀疏(实测 2 条),开启后打通内部调用链(实测 200 条);代价是慢一个数量级,故按场景开关(定向扫描关、全量扫描开)。
3. **子代理只读、监督者目标驱动**:子代理禁用一切写/执行工具,只能 `Read/Glob/Grep`,保证审计过程不改动被审代码;监督者用 `/goal` 跑到完成条件满足。
4. **证据分块 + 每 hunter 独立证据包**:大项目候选路径多,按字节/条数分块,按 hunter 相关性投递,避免单次上下文过载。
5. **攻击链关联**:在 finalize 阶段把单点发现(越权、命令注入)跨发现合成更高危的链(未认证 RCE)。

## 4. 主要数据产物(落在 `workspace/audit_<id>/`)

| 文件 | 内容 |
|---|---|
| `project/` | 解压/克隆后的被审源码 |
| `evidence/whitebox/*.json` | 入口、sink、候选路径(分块)、存储候选、未解析调用、覆盖度等静态分析产物 |
| `evidence/packages/` | 每个 hunter 的证据包 |
| `.claude/skills/`(job 工作目录下) | 每个子代理的 Skill(判定知识) |
| `supervisor-response.txt` | 监督者返回的原始 JSON 信封 |
| `findings.json` | 最终漏洞列表(含 severity / data_flow_path / verdict / reachability 等) |
| `job-meta.json` | 任务元信息 |

## 5. 历史演进 / 已移除技术(避免误解)

- **CodeQL**:早期版本用过,**已于 2026-06 移除**,改为 JavaParser 白盒。
- **claude-CLI 直调**:已移除,Claude 调用统一走 **Python Agent SDK sidecar**。
- **Joern(CPG + 真数据流 `reachableByFlows`)**:在 `feat/external-interface-deep-audit` 分支上(接口清单驱动的深度审计),**不在 master**。master 的"数据链路"是方法级调用可达 + 污点校验,不是 Joern 那种值级数据流。
